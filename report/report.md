# Вступ {#вступ .Заголовок-(без-номера)}

У сучасному світі інформаційні потоки відіграють надзвичайно важливу
роль, а ефективна обробка та публікація новин стає критичним завданням
для багатьох організацій. Однак у контексті великого потоку даних, який
надходить із різних джерел, виникає потреба в інструментах, що
дозволяють створювати релевантний і якісний контент на основі публічної
інформації, доступної в соціальних мережах та інших онлайн-ресурсах.

Актуальність даної роботи зумовлена необхідністю автоматизації процесів
збору інформації, її аналізу та формування новин. У зв\'язку з активним
використанням таких платформ, як Telegram, Instagram, Facebook та інших,
стає важливим забезпечення інтеграції з цими джерелами для збору
публічних даних та їхньої обробки відповідно до потреб редакторів і
журналістів.

На світовому рівні подібні проблеми вирішуються за допомогою
інструментів агрегування новин, але переважна більшість з них
фокусується на автоматичному розподілі новин без можливості створення
контенту на їхній основі. Пропонований веб-сервіс поєднує функціонал
агрегатора новин з редакторським інструментом для створення оригінальних
матеріалів на основі зібраної інформації.

Об\'єктом розробки є монолітний веб-застосунок, що дозволяє:

-- обирати джерела публічної інформації (канали в Telegram, сторінки в
Instagram та інші);

-- автоматизовано зтягувати дані з джерел за допомогою RSS \[2\] , API
або веб-скрапінгу;

-- надавати користувачам можливість редагувати отриману інформацію та
створювати власні новини;

-- публікувати готовий контент у зручному форматі.

Сфери застосування даного сервісу включають редакції новинних видань,
PR-агентства, блогерів та аналітичні центри. Завдяки інтеграції з
популярними соціальними мережами та підтримці багаторазового
використання даних, цей веб-застосунок може стати універсальним
інструментом для створення якісного контенту.

# ПЕРЕДПРОЄКТНЕ ОБСТЕЖЕННЯ ПРЕДМЕТНОЇ ОБЛАСТІ

## Аналіз предметної області 

Веб-додаток \[1\] -- це програмне забезпечення або програма, яку можна
відкрити за допомогою будь-якого браузера. Зовнішній інтерфейс веб
програми розробляється за допомогою таких мов програмування: HTML, CSS,
Javascript, які підтримуються на будь-якому браузері (Opera, Chrome,
Mozilla, Arc). У той час як для написання серверної частини (Back-end)
може використовуватися будь-яка інша мова програмування або фреймворк,
наприклад Python, PhP, Ruby, Java.

RSS \[2\] -- це родина XML \[7\] -форматів, що використовується для
публікації та постачання інформації, що часто змінюється, наприклад,
нових записів в блозі, заголовків новин, анонсів статей, зображень,
аудіо і відео матеріалів (в стандартизованому форматі). Документ в
стандарті RSS (який також інколи називають «стрічкою», «вебстрічкою» або
«каналом») складається з повного або часткового тексту і метаданих (дата
і авторство).

API \[3\] -- це набір інструкцій, які дозволяють різним програмам
спілкуватись між собою та обмінюватись даними.

Веб-скрапінг \[4\]- це процес автоматичного вилучення даних з
веб-сайтів, веб-сервісів і веб-додатків.

Предметна область даного проєкту -- це створення новин на основі
публічної інформації, доступної в соціальних мережах та інших
онлайн-ресурсах. У контексті сучасного розвитку інформаційних
технологій, соціальні мережі стали основним джерелом новин, які швидко
поширюються серед користувачів. Такі платформи, як Telegram, Instagram,
Facebook, Twitter, та інші, дозволяють отримувати оперативну інформацію
про події у світі, але її обробка та подальше використання для створення
якісного контенту вимагає значних зусиль.

На сьогодні існують численні сервіси для агрегування новин, які
автоматично збирають дані з різних джерел. Прикладами є Google News,
Feedly та Pocket. Вони надають можливість користувачам стежити за
новинами, але не підтримують редагування або створення новин на основі
зібраних даних. Це є суттєвим недоліком для редакцій і PR-агентств, які
потребують інструментів для адаптації контенту до потреб аудиторії.

У сфері ІТ реалізація роботи з соціальними мережами переважно базується
на використанні таких технологій:

-- RSS -- для отримання новин із джерел, що підтримують цей формат;

-- API -- для інтеграції з платформами, які надають відкриті прикладні
інтерфейси;

-- Веб-скрапінг -- для збору даних із джерел, що не мають інших методів
інтеграції, за умови дотримання правил використання.

Недоліки поточного стану:

-- Брак інструментів, які об\'єднують функціонал збору новин та їх
редагування;

-- Обмеження доступу до даних через зміни політик API платформ;

-- Велика кількість інформаційного шуму, що ускладнює аналіз даних.

Шляхи покращення ситуації:

-- Розробка універсального інструменту, що поєднує збір, обробку та
редагування новин;

-- Використання сучасних технологій для автоматизації процесів;

-- Забезпечення зручного інтерфейсу для користувачів із різним рівнем
підготовки.

У рамках даної курсової роботи обрано шлях розробки монолітного
веб-застосунку, що підтримує інтеграцію з основними джерелами інформації
через RSS, API та веб-скрапінг. Такий підхід дозволяє досягти
поставлених цілей, забезпечуючи ефективну обробку даних і створення
якісного контенту.

### Аналіз відомих програмних продуктів

Для аналізу відомих програмних продуктів у сфері роботи з новинами та їх
обробки було обрано кілька рішень, які частково реалізують функціонал,
схожий до описаного в технічному завданні: Feedly і NewsBlur. Нижче
наведено їхній опис та порівняльний аналіз із пропонованою розробкою.

#### Опис програмних продуктів:

-- Feedly є одним із найпопулярніших RSS-агрегаторів. Він дозволяє
користувачам збирати новини з різних джерел, організовувати їх у
тематичні стрічки та читати новини через зручний інтерфейс. Основний
недолік -- відсутність функцій редагування зібраного контенту або
створення нових публікацій;

-- NewsBlur -- це RSS-агрегатор, який дозволяє користувачам не лише
читати новини, але й використовувати автоматичні фільтри для
персоналізації потоку інформації. Проте NewsBlur, як і Feedly,
обмежується лише читанням контенту без можливості його редагування чи
створення нових матеріалів.

Отже, Feedly -- це популярний RSS-агрегатор, який дозволяє користувачам
підписуватися на новини та оновлення з різних джерел (блогів, новинних

надає користувачам можливість організовувати інформацію за категоріями,
використовувати теги для сортування та зберігати важливі статті для
подальшого перегляду.

NewsBlur -- це інший популярний RSS-агрегатор, який дозволяє
користувачам підписуватися на новини з різних джерел, організовувати їх
в одному місці та читати оновлення в реальному часі. NewsBlur має кілька
унікальних особливостей, які роблять його привабливим для користувачів,
які шукають більше контролю над своїм інформаційним потоком.

Основні переваги Feedly:

-- інтуїтивно зрозумілий інтерфейс і можливість доступу до вмісту на
різних пристроях;

-- підтримка інтеграції з іншими платформами, такими як Evernote,
Pocket, Twitter, LinkedIn;

-- користувачі можуть налаштовувати потоки новин, вибираючи тільки ті
теми або джерела, які їх цікавлять.

Недоліки:

-- обмежена функціональність у безкоштовній версії. Для доступу до
розширених можливостей (наприклад, більш детальної аналітики або
підтримки більше ніж 100 джерел) необхідно придбати преміум-підписку;

-- через велику кількість підписок користувачі можуть відчувати труднощі
з організацією та фільтрацією важливих новин.

Основні переваги NewsBlur:

-- система дозволяє \"навчити\" агрегатор, які новини або пости важливі
для користувача, а які можна ігнорувати, що покращує досвід читання;

-- NewsBlur підтримує функцію попереднього перегляду або повного
завантаження статей з деяких джерел;

-- можливість створення анонімних акаунтів і читання новин без
реєстрації або надання особистих даних;

-- NewsBlur здатен обробляти мультимедійний контент, зокрема зображення
та відео, без необхідності переходити за посиланнями.

Недоліки:

-- безкоштовна версія дозволяє підписатися лише на обмежену кількість
джерел (до 64), а також має обмежену функціональність для просунутих
користувачів;

-- хоча інтерфейс досить гнучкий, деякі новачки можуть знайти його трохи
складним для орієнтації, особливо з урахуванням великої кількості
налаштувань на застарілим дизайном.

### Аналіз відомих алгоритмічних та технічних рішень

Існуючі підходи до збору даних для формування новин можна розділити на
три основні категорії: використання RSS-каналів, інтеграція через API
платформ, а також веб-скрапінг. Кожен із цих підходів має свої переваги
та недоліки, що впливають на вибір інструментів для реалізації
програмного продукту.

а) використання RSS-каналів. RSS є одним із найпоширеніших способів
отримання новин із джерел, які підтримують цей формат;

Переваги:

> 1\) простота інтеграції;
>
> 2\) широка підтримка форматів багатьма новинними ресурсами;
>
> 3\) відсутність необхідності авторизації чи отримання спеціальних
> дозволів.

Недоліки:

> 1\) обмеження в обсязі доступної інформації (зазвичай доступні лише
> заголовки та короткий опис);
>
> 2\) не всі сучасні джерела підтримують RSS;
>
> 3\) відсутність інтерактивного оновлення даних у реальному часі.

б) інтеграція через API. Багато сучасних платформ, зокрема соціальні
мережі, надають API для доступу до своїх даних.

Переваги:

1\) доступ до структурованих даних із можливістю вибору полів і
параметрів;

2\) інтеграція в реальному часі з автоматичним оновленням;

3\) підтримка розширених функцій, наприклад, пошуку чи фільтрації даних.

Недоліки:

> 1\) політика використання API часто обмежує обсяг доступних даних
> (наприклад, через квоти);
>
> 2\) необхідність отримання ключів доступу та авторизації;
>
> 3\) часті зміни в політиці платформ, які можуть вимагати адаптації до
> нових умов.

в) веб-скрапінг. Цей підхід передбачає вилучення інформації
безпосередньо з веб-сторінок за допомогою спеціалізованих інструментів;

Переваги:

> 1\) можливість отримання даних із джерел, які не підтримують RSS чи
> API;
>
> 2\) повний контроль над тим, які саме дані отримуються.

Недоліки:

> 1\) ризик порушення політики конфіденційності або правил користування
> платформами;
>
> 2\) складність підтримки коду через можливі зміни у верстці
> веб-сторінок;
>
> 3\) затримка через необхідність аналізу та парсингу HTML-коду.

У рамках пропонованого рішення буде використовуватися комбінований
підхід. У першу чергу застосовуватимуться RSS-канали та API, а
веб-скрапінг -- лише у разі відсутності інших можливостей доступу до
інформації. Це дозволить досягти балансу між простотою реалізації,
відповідністю політикам платформ та гнучкістю отримання даних.

У процесі створення веб-застосунку для роботи з новинами було розглянуто
різні алгоритмічні та технічні рішення, які застосовуються для розробки
подібних систем. Особливу увагу приділено вибору архітектурних підходів,
здатних забезпечити високу ефективність, продуктивність і зручність
розробки.

а) архітектурні підходи. Розробка веб-додатків передбачає використання
різних архітектур залежно від вимог до масштабу, продуктивності та
функціональності. Основні підходи включають:

1)  монолітна архітектура: Цей підхід полягає у створенні застосунку як
    єдиного блоку, в якому всі компоненти безпосередньо взаємодіють між
    собою. Моноліт підходить для невеликих і середніх проєктів завдяки
    простоті реалізації;

2)  мікросервісна архітектура: Мікросервіси розділяють систему на
    незалежні компоненти, кожен з яких виконує конкретну функцію. Такий
    підхід дозволяє легко масштабувати систему та інтегрувати нові
    модулі;

3)  клієнт-серверна архітектура: Застосовується поділ на клієнтську
    частину (фронтенд) і серверну (бекенд), які взаємодіють через
    мережу. Це забезпечує гнучкість у розробці та підтримці;

4)  архітектура, керована подіями (Event-driven architecture): Цей
    підхід орієнтований на обробку подій у реальному часі, що дозволяє
    створювати реактивні та динамічні системи;

5)  сервіс-орієнтована архітектура (SOA): SOA будується на незалежних
    сервісах, які взаємодіють через стандартизовані інтерфейси, що
    забезпечує гнучкість і модульність системи.

б) технічні платформи для розгортання. Етап розгортання є важливим для
забезпечення доступності застосунку для кінцевих користувачів.
Розглянуто кілька сучасних рішень:

1)  хмарні сервіси: Платформи на зразок Heroku, Vercel і Netlify
    пропонують простоту використання та автоматизоване розгортання, що
    робить їх ідеальними для невеликих проєктів;

2)  великі хмарні платформи: Amazon Web Services (AWS), Google Cloud
    Platform (GCP) і Microsoft Azure забезпечують високу гнучкість,
    масштабованість і широкий набір інструментів, таких як бази даних і
    хмарне зберігання. AWS популярний завдяки широким можливостям, а GCP
    зручний для команд, які працюють із машинним навчанням чи аналізом
    даних;

3)  віртуальні приватні сервери (VPS): DigitalOcean, Linode і Hetzner
    дозволяють самостійно налаштовувати сервери, що забезпечує контроль,
    але вимагає базових знань адміністрування;

4)  контейнеризація та оркестрація: Docker забезпечує ізоляцію
    середовища для застосунків, а Kubernetes дозволяє ефективно
    масштабувати системи. Ці рішення ідеальні для складних проєктів, але
    вимагають досвіду в DevOps;

5)  власні сервери: Найгнучкіше рішення, яке дає повний контроль над
    конфігурацією, але потребує значних технічних знань.

Кожне з описаних рішень має свої переваги та недоліки, що слід
враховувати залежно від вимог до проєкту. Для створення веб-застосунку з
оптимальним поєднанням функціональності, масштабованості та простоти
реалізації може бути обрано монолітну архітектуру з подальшим
використанням хмарних платформ або VPS для розгортання.

  Flow of Events   Користувач відкриває опубліковану новину. Користувач
                   натискає кнопку \"Поскаржитися\". Користувач заповнює
                   форму скарги, вказуючи причину (наприклад, некоректний
                   контент, порушення авторських прав). Користувач
                   натискає кнопку \"Надіслати\". Система зберігає скаргу
                   для обробки адміністратором (UC-15). Користувач отримує
                   підтвердження про надсилання скарги.

  Extension        Якщо форма скарги заповнена некоректно (наприклад, не
                   вказано причину), відображається повідомлення про
                   помилку. Користувач повертається до заповнення форми.

  Post-Condition   Скаргу збережено в системі, адміністратор отримує
                   сповіщення для обробки.

## Розроблення нефункціональних вимог

Нефункціональні вимоги визначають якісні характеристики веб-сервісу для
формування новин на основі публічних сторінок у соціальних мережах. Вони
забезпечують стабільну, безпечну та зручну роботу системи для гостей,
користувачів і адміністраторів, а також підтримують обробку великих
обсягів даних із соціальних мереж. Вимоги поділено на категорії, кожна з
яких описує специфічні аспекти функціонування системи.

### Продуктивність

Система повинна забезпечувати швидку обробку запитів користувачів і
ефективну роботу з даними, включаючи збір, фільтрацію, редагування та
публікацію контенту. Час відповіді на запити має бути мінімальним навіть
при високому навантаженні, щоб забезпечити комфортну роботу для всіх
користувачів.

- Час відповіді на запит (авторизація, пошук джерел, публікація новин)
  не повинен перевищувати 2 секунди для 95% запитів при навантаженні до
  1000 одночасних користувачів.

- Збір даних із джерел (RSS, API, веб-скрапінг) має виконуватися не
  довше 10 секунд на одне джерело при стандартному обсязі даних (до 10
  МБ).

### Безпека

Система повинна гарантувати захист даних користувачів, контенту та
процесів авторизації від несанкціонованого доступу, атак і витоків
інформації.

-- Усі дані користувачів (email, паролі) шифруються за допомогою
протоколу HTTPS і алгоритму хешування паролів (наприклад, bcrypt).

- JWT-токени \[9\] для авторизації мають обмежений час дії (не більше 24
  годин) і захищені від модифікації.

- Система запобігає SQL-ін'єкціям \[10\] і XSS-атакам шляхом валідації
  та екранування всіх вхідних даних.

### Доступність

Система повинна бути доступною для користувачів у будь-який час із
мінімальними перебоями в роботі, забезпечуючи стабільний доступ до всіх
функцій сервісу.

- Рівень доступності системи становить не менше 99.9% часу на місяць
  (uptime).

- У разі збою система автоматично перенаправляє запити на резервні
  сервери протягом 30 секунд.

- Сервіс підтримує безперебійну роботу при піковому навантаженні до 5000
  одночасних користувачів.

### Масштабованість

Система повинна підтримувати зростання кількості користувачів, джерел і
обсягу оброблюваних даних без значного зниження продуктивності.

- Система здатна масштабуватися горизонтально шляхом додавання нових
  серверів для обробки запитів.

- База даних підтримує розподілене зберігання даних для обробки до 1 ТБ
  контенту.

- Система забезпечує стабільну роботу при додаванні до 100 нових джерел
  на день.

### Зручність використання

Інтерфейс системи має бути інтуїтивно зрозумілим і зручним для
користувачів із різним рівнем технічної підготовки, включаючи гостей,
користувачів і адміністраторів.

-- Час, необхідний для виконання основних дій (реєстрація, додавання
джерела, створення новини), не перевищує 2 хвилин для нового
користувача.

-- Інтерфейс адаптивний і коректно відображається на пристроях із
роздільною здатністю від 320x480 до 1920x1080 пікселів. --

### Надійність

Система повинна забезпечувати стабільну роботу без критичних збоїв і
втрати даних, включаючи збереження контенту, джерел і скарг.

-- У разі збою система автоматично створює резервні копії бази даних
кожні 24 години.

-- Помилки обробки даних (наприклад, некоректне джерело) не призводять
до припинення роботи сервісу.

### Сумісність

Система повинна бути сумісною з різними платформами, браузерами та
операційними системами, щоб забезпечити доступність для широкої
аудиторії.

-- Сервіс підтримує сучасні браузери актуальних версій (Google Chrome,
Mozilla Firefox, Safari, Microsoft Edge).

-- Система сумісна з операційними системами Windows 10+, macOS 11+,
Linux (Ubuntu 20.04+), Android 10+, iOS 15+.

-- API системи відповідає стандартам REST і підтримує інтеграцію з
зовнішніми сервісами (наприклад, Telegram, Instagram).

### Портативність

Система повинна бути легко переносимою на нові апаратні або програмні
платформи без значних змін у коді.

-- Код системи контейнеризовано (наприклад, за допомогою Docker) для
швидкого розгортання на різних серверах.

-- Система підтримує розгортання в хмарних середовищах (AWS, Google
Cloud, Azure) із мінімальними налаштуваннями.

## Постановка задачі

та інших соціальних мереж, аналізувати їх, редагувати контент і
створювати на його основі новини для подальшої публікації.

Цілі:

- Забезпечити зручний механізм інтеграції з джерелами інформації за
  допомогою RSS, API або веб-скрапінгу;

- Надати функціональність для редагування зібраного контенту та
  створення нових матеріалів;

- Забезпечити можливість публікації створеного контенту на веб-сторінці
  новин;

Задачі для розв'язання:

- Реалізувати механізм збору даних із джерел, використовуючи відповідні
  технології;

> а) RSS для структурованих джерел;
>
> б) API для платформ, що надають програмні інтерфейси;
>
> в) веб-скрапінг для платформ без RSS або API (з урахуванням політики
> використання).

-- Розробити редактор контенту, який дозволяє:

> а) Змінювати текстові й графічні матеріали;
>
> б)Додавати нові елементи (зображення, заголовки).

-- Реалізувати веб-інтерфейс для взаємодії з користувачами:

> а) зручний UI для перегляду, редагування та публікації матеріалів;
>
> б) функціонал пошуку та фільтрації новин.

-- Забезпечити надійну архітектуру для зберігання даних:

> а) розробити структуру бази даних для зберігання контенту;
>
> б) реалізувати механізми безпечного доступу до даних.

-- Тестування і оптимізація роботи системи:

> а) перевірка роботи механізмів збору даних із різних джерел;
>
> б) забезпечення швидкої та стабільної роботи веб-застосунку.

Очікуваним результатом є функціональний веб-сервіс, який дозволяє
редакторам створювати якісні новини на основі інформації, зібраної з
різних джерел, і публікувати їх на новинній сторінці.

У розділі 2 виконано комплексний аналіз і розробку вимог до веб-сервісу
для формування новин на основі публічних сторінок у соціальних мережах,
що дозволило сформувати чітке бачення функціоналу та характеристик
системи. Робота охопила чотири ключові підпункти, кожен із яких вніс
вагомий внесок у визначення специфікацій проєкту.

У підпункті 2.1 описано головний функціонал сервісу, який полягає в
автоматизованому зборі, обробці, редагуванні та публікації новин із
соціальних мереж. Розроблено діаграму варіантів використання (Рисунок
2.1), що відображає взаємодію акторів (Гість, Користувач, Адміністратор)
із системою через 16 варіантів використання (UC-01--UC-16). Детальний
опис кожного варіанту використання представлено в таблицях 2.1--2.16, що
включають цілі, акторів, передумови, потік подій, розширення та
постумови, забезпечуючи повне розуміння функціональних сценаріїв.

Підпункт 2.2 присвячено аналізу системних вимог, де визначено мінімальну
та рекомендовану конфігурацію технічних засобів для клієнтських
пристроїв.

У підпункті 2.3 розроблено функціональні вимоги, структуровані за
модулями системи: авторизація, управління джерелами, обробка контенту,
скарги та адміністрування. Загальна модель вимог (Таблиця 2.17) включає
21 вимогу (FR-1--FR-21) із пріоритетами та ризиками. Перелік вимог
представлено в таблиці 2.16, а детальний опис -- у таблиці 2.18. Матриця
трасування (Рисунок 2.3) пов'язує вимоги з UC, забезпечуючи відстеження
їх реалізації.

Підпункт 2.4 зосереджено на нефункціональних вимогах, структурованих за
категоріями: продуктивність, безпека, доступність, масштабованість,
зручність використання, надійність, сумісність і портативність
(2.4.1--2.4.8). Кожна категорія включає конкретні вимоги, такі як час
відповіді до 2 секунд, шифрування даних через HTTPS, 99.9% доступності
та підтримка сучасних браузерів, що забезпечують якісну роботу сервісу.

Таким чином, розділ 2 створив міцну основу для подальшої розробки
веб-сервісу, чітко визначивши його функціональні та нефункціональні
характеристики, а також системні вимоги. Отримані результати
забезпечують повне розуміння потреб системи та її взаємодії з
користувачами, що є ключовим для успішної реалізації проєкту.

За результатами розділу сформовано технічне завдання на розробку
програмного забезпечення.

# КОНСТРУЮВАННЯ ТА РОЗРОБЛЕННЯ ПРОГРАМНОГО ЗАБЕЗПЕЧЕННЯ

## Архітектура програмного забезпечення

Архітектура веб-сервісу для формування новин на основі публічних
сторінок у соціальних мережах розроблена як монолітна з використанням
принципів Domain-Driven Design \[8\] (DDD). Монолітна архітектура обрана
для спрощення розробки, розгортання та підтримки системи на початкових
етапах, враховуючи обмежену складність проєкту та потребу в
централізованому управлінні функціоналом. DDD застосовується для
структуризації коду навколо бізнес-доменів (авторизація, управління
джерелами, обробка контенту, скарги, адміністрування), що забезпечує
чітке розділення відповідальностей і полегшує майбутнє масштабування.

Система включає єдиний серверний додаток, який обробляє всі запити
користувачів, та декілька скриптів, які інтегруються з зовнішніми
джерелами даних (RSS, API соціальних мереж, веб-скрапінг). Серверний
додаток і взаємодіє з базою даних для зберігання контенту, джерел і
скарг. Архітектура представлена через діаграми C4 Model на трьох рівнях:
System Context (L1), Container (L2) і Component (L3), які деталізують
взаємодію системи з зовнішніми акторами, її основні контейнери та
внутрішні компоненти.

Діаграма рівня L1 (System Context) зображена на рис. 3.1. На високому
рівні веб-сервіс представлено як єдину систему, що взаємодіє з трьома
типами акторів та двома зовнішніми системами. Гості переглядають
сторінку привітання, реєструються або авторизуються, користувачі додають
джерела, обробляють контент, публікують новини та залишають скарги, а
адміністратори керують джерелами й обробляють скарги. Система отримує
дані з соціальних мереж, таких як Telegram та Instagram, через REST API,
а також із RSS-каналів у форматі XML \[7\] або JSON. Ця структура
відображає основні взаємодії через HTTPS для акторів і API для зовнішніх
джерел, забезпечуючи просте розуміння контексту системи.

Внутрішня структура веб-сервісу (рис. 3.2) розкриває три основні
контейнери: веб-додаток, серверний додаток і база даних. Веб-додаток,
побудований на Vue.js із TypeScript і стилізований через Tailwind CSS,
забезпечує адаптивний інтерфейс для всіх акторів, дозволяючи виконувати
дії через HTTPS-запити. Серверний додаток, реалізований на NestJS \[6\]
(Node.js, TypeScript), обробляє бізнес-логіку, інтегрується з зовнішніми
джерелами та взаємодіє з базою даних через SQL. База даних на PostgreSQL
зберігає інформацію про користувачів, джерела, контент, новини та
скарги. Веб-додаток комунікує із серверним через REST API, а серверний
додаток отримує дані від соціальних мереж і RSS-каналів, використовуючи
відповідні протоколи.

Діаграма рівня L3 (Component) деталізує внутрішню структуру серверного
додатка, розбиваючи його на компоненти відповідно до принципів DDD.
Серверний додаток деталізовано на компоненти, структуровані за
принципами DDD, кожен із яких відповідає окремому домену. Контролер API,
побудований на NestJS \[6\] Controllers, обробляє вхідні REST-запити та
делегує їх відповідним сервісам. Сервіс авторизації управляє
реєстрацією, верифікацією email, авторизацією та виходом, використовуючи
JWT-токени. Сервіс джерел відповідає за додавання, перевірку та пошук
джерел новин. Сервіс контенту забезпечує збір, фільтрацію, редагування,
створення та публікацію новин. Сервіс скарг обробляє залишення та
розгляд скарг, а сервіс адміністрування надає функції управління
джерелами для адміністраторів. Інтеграційний модуль, що використовує
бібліотеку axios, забезпечує взаємодію із зовнішніми джерелами через
REST API та парсинг RSS у форматі XML \[7\] /JSON. Усі сервіси
взаємодіють із базою даних через TypeORM, а сервіси джерел і контенту
використовують інтеграційний модуль для збору даних.

## Важливі архітектурні рішення

Інтеграція із зовнішніми системами здійснюється через веб-скрапінг для
соціальних мереж Instagram, Twitter/X. RSS-канали обробляються через
HTTP-запити з парсингом XML \[7\], що є стандартом для новинних джерел.
Ці протоколи обрано через їхню універсальність і підтримку безпеки. Для
виконання нефункціональних вимог продуктивність забезпечується
асинхронною обробкою в NestJS \[6\] і кешуванням запитів через Redis, що
гарантує час відповіді до 2 секунд. Безпека досягається завдяки
JWT-токенам, хешуванню паролів через bcrypt і захисту від ін'єкцій через
Drizzle ORM. Доступність на рівні 99.9% підтримується хмарним
розгортанням із резервними серверами, а масштабованість -- через
контейнеризацію (Docker) і шардування PostgreSQL. Зручність використання
гарантується адаптивним інтерфейсом Vue.js і локалізацією, а надійність
-- щоденним резервним копіюванням і обробкою помилок у NestJS \[6\].
Сумісність із сучасними браузерами та ОС забезпечується стандартами
Vue.js, а портативність -- контейнеризацією та TypeScript. PostgreSQL
обрано як основну базу даних через її підтримку реляційних даних і
масштабованості, що ідеально відповідає структурі даних системи.

## Обґрунтування вибору засобів розробки

Для розробки веб-сервісу формування новин на основі публічних сторінок у
соціальних мережах обрано технологічний стек, що включає TypeScript як
основну мову програмування, Vue.js для клієнтської частини, NestJS \[6\]
для серверної частини, PostgreSQL як базу даних, S3 сховище для файлів
та Redis як in-memory сховище, а також допоміжні бібліотеки та
інструменти, такі як Tailwind CSS, Drizzle, Docker і Visual Studio Code.
Вибір цього стеку зумовлений вимогами до продуктивності, безпеки,
масштабованості, зручності використання та сумісності, визначеними у
нефункціональних вимогах, а також потребою в швидкій розробці та
підтримці монолітної архітектури з принципами Domain-Driven Design.

TypeScript обрано як основну мову програмування завдяки строгій
типізації, яка зменшує кількість помилок під час розробки та полегшує
підтримку коду, що критично для складних бізнес-доменів, таких як
авторизація, обробка контенту та адміністрування. Порівняно з
JavaScript, TypeScript забезпечує кращу масштабованість проєкту та
інтеграцію з сучасними фреймворками, такими як Vue.js і NestJS \[6\].
Альтернативи, такі як Python або Java, були відхилені: Python, хоча й
популярний для веб-розробки (наприклад, Django), менш ефективний для
асинхронної обробки великої кількості запитів, а Java потребує більше
ресурсів і ускладнює швидке розгортання. TypeScript, завдяки своїй
універсальності та підтримці Node.js, ідеально відповідає потребам
веб-сервісу.

Для клієнтської частини Vue.js обрано через його легкість, гнучкість і
швидке створення адаптивних інтерфейсів, що відповідає вимозі зручності
використання. Vue.js у комбінації з TypeScript і Pinia для управління
станом забезпечує модульну структуру та інтуїтивний API, що спрощує
розробку складних форм, таких як додавання джерел чи редагування новин.
Порівняно з React, Vue.js має менший поріг входження та кращу
продуктивність для невеликих і середніх проєктів, тоді як Angular був
відхилений через надмірну складність для монолітної системи. Tailwind
CSS використано для стилізації завдяки його утилітарному підходу, який
дозволяє швидко створювати адаптивні дизайни без написання великої
кількості CSS-коду. Альтернативи, такі як Bootstrap, менш гнучкі для
кастомізації, а чисте CSS потребує більше часу на розробку.

Серверна частина реалізована на NestJS \[6\], фреймворку для Node.js,
який підтримує TypeScript і модульну архітектуру, ідеальну для DDD.
NestJS \[6\] забезпечує вбудовану підтримку REST API, асинхронну обробку
запитів і захист від ін'єкцій через Pipes, що відповідає вимогам
продуктивності та безпеки. У порівнянні з Express.js, NestJS \[6\]
пропонує кращу структуризацію коду та вбудовані інструменти для
тестування, тоді як Fastify, хоча й швидший, менш зрілий для складних
проєктів. Django (Python) був відхилений через меншу ефективність у
реальному часі, а Spring (Java) -- через складність розгортання. Для
взаємодії з базою даних використано Drizzle, який забезпечує зручну
роботу з PostgreSQL і підтримує транзакції, необхідні для цілісності
даних. Альтернатива, Prisma, хоча й сучасна, менш гнучка для складних
реляційних моделей.

PostgreSQL обрано як базу даних через її підтримку реляційних даних,
транзакцій, індексів і масштабованості, що ідеально відповідає потребам
зберігання користувачів, джерел, публікацій, новин і скарг. Порівняно з
MongoDB, PostgreSQL краще забезпечує транзакційну консистентність,
необхідну для обробки скарг і авторизації, тоді як MySQL був відхилений
через меншу підтримку JSONB для неструктурованого контенту. Redis
використано як допоміжну систему для кешування запитів до зовнішніх API,
що підвищує продуктивність.

Для інтеграції із зовнішніми джерелами використано бібліотеку axios, яка
забезпечує просту та надійну роботу з REST API соціальних мереж
(Telegram, Instagram). Порівняно з fetch, axios має кращу підтримку
обробки помилок і конфігурації. Для парсингу RSS-каналів обрано xml2js,
який ефективно обробляє XML \[7\] і JSON, тоді як альтернатива,
feedparser, менш сумісна з TypeScript. Ці бібліотеки відповідають
вимогам універсальності та безпеки.

Розробка ведеться в інтегрованому середовищі Visual Studio Code, яке
підтримує TypeScript, ESLint, Prettier і плагіни для роботи з Vue.js і
NestJS \[6\]. VS Code обрано через його легкість, широку підтримку
розширень і безкоштовність, порівняно з платними IDE, такими як
WebStorm, які надають подібний функціонал, але потребують додаткових
витрат. Для тестування використано Jest, що забезпечує швидке написання
юніт-тестів для NestJS \[6\] і Vue.js, а для контейнеризації -- Docker,
який гарантує портативність і сумісність із хмарними платформами.
CI/CD-пайплайни налаштовано через GitHub Actions для автоматизації
розгортання.

Таким чином, обраний технологічний стек -- TypeScript, Vue.js, NestJS
\[6\], PostgreSQL, Tailwind CSS, axios, Drizzle, Docker і VS Code --
забезпечує оптимальний баланс між продуктивністю, безпекою,
масштабованістю та швидкістю розробки. Порівняльний аналіз альтернатив
підтвердив, що ці засоби найкраще відповідають вимогам проєкту,
дозволяючи ефективно реалізувати монолітну архітектуру з DDD і
забезпечити майбутнє розширення функціоналу.

## Конструювання програмного забезпечення

[]{#_Toc215602994 .anchor}3.4.1 Опис оригінальних алгоритмів

У процесі розробки веб-сервісу було реалізовано кілька оригінальних
алгоритмів, що забезпечують ефективну роботу системи збору та обробки
даних із різних джерел. Нижче наведено опис основних алгоритмічних
рішень.

а) Алгоритм динамічного розрахунку пріоритетів джерел на основі
активності користувачів.

Для оптимізації навантаження на систему та забезпечення актуальності
даних реалізовано алгоритм динамічного розрахунку пріоритетів джерел
новин. Алгоритм базується на аналізі активності користувачів, які
підписані на конкретне джерело, протягом заданого часового
вікна.Алгоритм працює наступним чином:

> 1\) Етап збору статистики: Система аналізує кількість активних
> користувачів для кожного джерела за заданий період (наприклад, останні
> 24 години). Активність визначається на основі переглядів контенту з
> конкретного джерела.
>
> 2\) Етап розрахунку пріоритету: На основі кількості активних
> користувачів визначається пріоритет джерела за наступною шкалою:

- Пріоритет 1: 100 і більше активних користувачів;

- Пріоритет 2: від 50 до 99 активних користувачів;

- Пріоритет 3: від 20 до 49 активних користувачів;

- Пріоритет 4: від 10 до 19 активних користувачів;

- Пріоритет 5: від 5 до 9 активних користувачів;

- Пріоритет 6: від 2 до 4 активних користувачів;

- Пріоритет 7: 1 активний користувач;

- Пріоритет 10: відсутність активних користувачів (джерело не
  обробляється).

> 3\) Етап розрахунку інтервалу оновлення: На основі пріоритету
> визначається інтервал між зборами даних:

- 3 хвилини для джерел з пріоритетом 1;

- 5 хвилин для джерел з пріоритетом 2;

- 10 хвилин для джерел з пріоритетом 3;

- 15 хвилин для джерел з пріоритетом 4;

- 30 хвилин для джерел з пріоритетом 5;

- 60 хвилин для джерел з пріоритетом 6;

- 120 хвилин для джерел з пріоритетом 7;

- 0 хвилин для джерел з пріоритетом 10 (збір не виконується).

> 4\) Етап планування завдань: Система створює або оновлює повторювані
> завдання в черзі оркестратора з розрахованим пріоритетом та інтервалом
> виконання.Алгоритм виконується автоматично через cron-завдання кожні 5
> хвилин, що забезпечує актуальність пріоритетів та адаптацію системи до
> змін у активності користувачів.

б) Алгоритм оркестрації збору даних з автоматичним маршрутизуванням.

Для забезпечення ефективної обробки різних типів джерел реалізовано
алгоритм оркестрації, який автоматично маршрутизує завдання збору даних
до відповідних черг обробки. Алгоритм складається з наступних етапів:

> 1\) Етап завантаження метаданих джерела: Система завантажує з бази
> даних повну інформацію про джерело, включаючи URL, тип джерела, стан
> (активне, призупинене, заблоковане), курсор останнього збору та
> додаткові метадані.
>
> 2\) Етап перевірки стану джерела: Виконується валідація стану джерела:

- Якщо джерело заблоковане (banned), обробка припиняється;

- Якщо джерело призупинене (paused), обробка призупиняється;

- Якщо джерело має статус помилки, виконується спроба збору (можливе
  відновлення).

> 3\) Етап витягування ідентифікатора: З URL джерела витягується
> зовнішній ідентифікатор (externalId) залежно від типу джерела:

- Для Instagram: витягується ім\'я користувача з URL (наприклад,
  <https://www.instagram.com/nasa/> = nasa);

- Для Telegram: витягується назва каналу або користувача
  (наприклад, https://t.me/channel = channel);

- Для Twitter/X: витягується ім\'я користувача
  (наприклад, https://twitter.com/nasa = nasa);

- Для RSS: використовується повний URL як ідентифікатор.

> 4) Етап збагачення даних завдання: Створюється об\'єкт завдання з
> повною інформацією, включаючи:

- Ідентифікатор джерела;

- Тип джерела;

- Зовнішній ідентифікатор;

- Курсор останнього збору (для пагінації);

- Ліміт кількості записів для збору;

- Пріоритет завдання;

- Метадані (час планування, ідентифікатор завдання оркестратора).

> 5) Етап маршрутизації: Завдання направляється до відповідної черги
> обробки залежно від типу джерела:

- Instagram потрапляє в чергу sources.instagram-fetcher;

- Telegram потрапляє в чергу sources.telegram-fetcher;

- RSS потрапляє в чергу sources.rss-fetcher;

- Twitter потрапляє в чергу sources.twitter-fetcher.

Алгоритм забезпечує централізоване управління процесом збору даних та
автоматичне розподілення навантаження між різними обробниками.

в) Алгоритм витягування ідентифікаторів з URL для різних типів
соціальних мереж.

Для уніфікації роботи з різними типами джерел реалізовано алгоритм
автоматичного витягування ідентифікаторів з URL за допомогою регулярних
виразів.Алгоритм використовує наступні правила парсингу:

> 1) Instagram: Використовується регулярний
> вираз /instagram\\.com\\/(\[\^/?#\]+)/ для витягування імені
> користувача з URL. Приклад: https://www.instagram.com/nasa/ = nasa.
>
> 2) Telegram: Використовується регулярний
> вираз /t\\.me\\/(\[\^/?#\]+)/ для витягування назви каналу або
> користувача. Приклад: https://t.me/channel = channel.
>
> 3) Twitter/X: Використовується регулярний
> вираз /(?:twitter\\.com\|x\\.com)\\/(\[\^/?#\]+)/ для підтримки обох
> доменів (twitter.com та x.com).
> Приклад: https://twitter.com/nasa або https://x.com/nasa = nasa.
>
> 4) RSS: Повний URL використовується як ідентифікатор без додаткової
> обробки.У разі невдалого парсингу алгоритм повертає оригінальний URL
> як резервний варіант, що забезпечує надійність роботи системи.

г) Алгоритм асинхронної обробки завдань через систему черг.

Для забезпечення масштабованості та надійності системи реалізовано
алгоритм асинхронної обробки завдань на основі черг BullMQ, що працює
поверх Redis. Алгоритм включає наступні компоненти:

> 1) Система пріоритетів: Завдання обробляються відповідно до їх
> пріоритету, що дозволяє спочатку обробляти найважливіші джерела.
>
> 2) Повторювані завдання: Для кожного джерела створюється повторюване
> завдання з динамічним інтервалом, який автоматично оновлюється на
> основі активності користувачів.
>
> 3) Обробка результатів: Після збору даних результати направляються до
> централізованої черги результатів, де виконується:

- Збереження зібраних публікацій у базі даних;

- Оновлення кешу в Redis;

- Оновлення метаданих джерела (курсор, час останнього збору).

> 4) Обробка помилок: У разі помилки під час збору даних система
> автоматично логує помилку та оновлює статус джерела, не припиняючи
> роботу інших завдань. Цей підхід забезпечує високу продуктивність
> системи та дозволяє обробляти велику кількість джерел одночасно без
> блокування основних процесів.

3.4.2 Опис структур даних та програмних структур

Система використовує структуровані типи даних для забезпечення
типобезпеки та консистентності. Основні структури даних реалізовані
через TypeScript типи та Drizzle ORM схеми.

а) Структури даних для роботи з джерелами

Для роботи з джерелами новин використовуються наступні структури:

> 1\) OrchestratorJobData -- структура даних для завдань оркестратора,
> що містить:

- sourceId: унікальний ідентифікатор джерела;

- priority: пріоритет завдання (від 1 до 10);

- scheduledBy: джерело планування (cron або manual).

> 2\) CollectorJobData -- структура даних для завдань збору, що розширює
> OrchestratorJobData та містить:

- sourceType: тип джерела (telegram, instagram, rss, twitter);

- externalId: зовнішній ідентифікатор (ім\'я користувача, URL каналу);

- cursor: курсор для пагінації;

- limit: максимальна кількість записів для збору;

- metadata: додаткові метадані (час планування, ідентифікатор завдання
  оркестратора).

б) Структури даних для зібраного контенту

Для збереження зібраного контенту використовується структура
FetchedPost, що містить:

> 1\) externalId: зовнішній ідентифікатор публікації;
>
> 2\) title: заголовок публікації;
>
> 3\) content: вміст у форматі JSON;
>
> 4\) mediaUrls: масив URL зображень та відео;
>
> 5\) publishedAt: дата публікації;
>
> 6\) metadata: додаткові метадані (автор, теги тощо).

в) Програмні структури модулів

Система організована за принципами Domain-Driven Design з наступними
основними модулями:

> 1\) Модуль авторизації (auth): відповідає за реєстрацію, верифікацію
> email, авторизацію через JWT-токени та OAuth-інтеграцію.
>
> 2\) Модуль користувачів (users): управління користувачами, ролями та
> профілями.
>
> 3\) Модуль джерел (sources): управління джерелами новин, включаючи:

- SourcesPriorityCalculatorService: розрахунок пріоритетів;

- SourcesOrchestratorService: оркестрація збору даних;

- SourcesValidationService: валідація джерел;

- SourcesResultService: обробка результатів збору.

> 4\) Модуль публікацій (raw-posts): зберігання та управління зібраними
> публікаціями.
>
> 5\) Модуль статей (articles): створення, редагування та публікація
> новин на основі зібраного контенту.
>
> 6\) Модуль скарг (complaints): обробка скарг користувачів на контент.

г) Зв\'язки між таблицями

База даних використовує наступні зв\'язки між таблицями:

1)  Один користувач може додати багато джерел, що реалізовано через поле
    added_by в таблиці Sources, яке посилається на таблицю Users. При
    видаленні користувача значення встановлюється як порожнє.

2)  Один користувач може створити багато статей, що реалізовано через
    поле author_id в таблиці Articles, яке посилається на таблицю Users.
    При видаленні користувача видаляються всі його статті.

3)  Одне джерело може містити багато публікацій, що реалізовано через
    поле source_id в таблиці RawPosts, яке посилається на
    таблицю Sources. При видаленні джерела видаляються всі пов\'язані
    публікації.

4)  Користувачі та джерела пов\'язані багато-до-багатьох через таблицю
    UserSources, яка містить посилання на обидві таблиці. При видаленні
    користувача або джерела видаляються відповідні записи в таблиці
    зв\'язків.

5)  Статті та публікації пов\'язані багато-до-багатьох через таблицю
    ArticleRawPosts, яка містить посилання на обидві таблиці. При
    видаленні статті або публікації видаляються відповідні записи в
    таблиці зв\'язків.

6)  Один користувач може подати багато скарг, що реалізовано через поле
    reported_by в таблиці Complaints, яке посилається на таблицю Users.
    При видаленні користувача видаляються всі його скарги.

7)  Один адміністратор може обробити багато скарг, що реалізовано через
    поле resolved_by в таблиці Complaints, яке посилається на
    таблицю Users. При видаленні адміністратора значення встановлюється
    як порожнє.

8)  Один користувач може мати один OAuth акаунт, що реалізовано через
    поле user_id в таблиці OAuthAccounts, яке посилається на таблицю
    Users та має унікальне обмеження. При видаленні користувача
    видаляється пов\'язаний OAuth акаунт.

9)  Один користувач може мати один токен оновлення, що реалізовано через
    поле user_id в таблиці RefreshTokens, яке посилається на таблицю
    Users та має унікальне обмеження. При видаленні користувача
    видаляється пов\'язаний токен.

10) Один користувач може мати одну верифікацію електронної пошти, що
    реалізовано через поле id в таблиці EmailVerifications, яке
    посилається на таблицю Users та використовується як первинний ключ.
    При видаленні користувача видаляється пов\'язана верифікація.

3.4.4 Опис структур даних та програмних структур

NestJS -- фреймворк для Node.js, використовується для серверної частини.
Надає модульну архітектуру, вбудовану підтримку TypeScript, декоратори
для контролерів, сервісів та middleware, а також інтеграцію з різними
бібліотеками. Дозволяє структурувати код за принципами Domain-Driven
Design та забезпечує масштабованість.

TypeScript -- мова програмування, що розширює JavaScript статичною
типізацією. Використовується для серверної та клієнтської частин.
Забезпечує раннє виявлення помилок, покращує читабельність коду та
підтримку, спрощує рефакторинг та інтеграцію з IDE.

Drizzle ORM -- об\'єктно-реляційний маппер для TypeScript.
Використовується для типобезпечної роботи з PostgreSQL. Надає міграції,
генерацію типів зі схеми БД, підтримку транзакцій та складних запитів.
Альтернатива TypeORM та Prisma.

PostgreSQL -- реляційна СУБД. Використовується як основна база даних для
зберігання користувачів, джерел, публікацій, статей та скарг. Підтримує
транзакції, індекси, JSONB для неструктурованих даних та масштабування.

Redis -- in-memory сховище даних. Використовується для кешування
запитів, сесій та як брокер черг для BullMQ. Забезпечує швидкий доступ
до даних та підтримку асинхронної обробки завдань.

BullMQ -- бібліотека для роботи з чергами завдань на базі Redis.
Використовується для асинхронної обробки збору даних, пріоритизації
завдань, повторюваних завдань та обробки помилок. Забезпечує надійність
та масштабованість системи черг.

Nuxt 3 -- фреймворк на базі Vue.js для створення веб-додатків.
Використовується для клієнтської частини. Надає серверний рендеринг,
автоматичну маршрутизацію, оптимізацію продуктивності та інтеграцію з
TypeScript.

Vue 3 -- JavaScript-фреймворк для створення інтерфейсів.
Використовується як основа Nuxt 3. Надає реактивність, компонентну
архітектуру, Composition API та оптимізацію продуктивності.

Tailwind CSS -- утилітарний CSS-фреймворк. Використовується для
стилізації інтерфейсу. Дозволяє швидко створювати адаптивні дизайни
через утилітарні класи без написання власного CSS.

Pinia -- бібліотека для управління станом у Vue-додатках.
Використовується для зберігання глобального стану додатку, даних
користувача та налаштувань. Альтернатива Vuex з простішим API та кращою
підтримкою TypeScript.

EditorJS -- блоковий редактор контенту. Використовується для створення
та редагування статей. Надає модульну архітектуру, підтримку різних
типів блоків (текст, зображення, списки, таблиці) та експорт у форматі
JSON.

Zod -- бібліотека валідації схем для TypeScript. Використовується для
валідації даних на сервері та клієнті. Забезпечує типобезпечну
валідацію, автоматичну генерацію типів зі схем та детальні повідомлення
про помилки.

bcrypt -- бібліотека для хешування паролів. Використовується для
безпечного зберігання паролів користувачів. Застосовує алгоритм bcrypt з
автоматичною генерацією солі та налаштовуваною складністю.

JWT (JSON Web Tokens) -- стандарт для безпечної передачі інформації між
сторонами. Використовується для авторизації користувачів через токени
доступу та оновлення. Забезпечує безсерверну авторизацію та
масштабованість.

Swagger -- інструмент для документування REST API. Використовується
через \@nestjs/swagger для автоматичної генерації документації API на
основі декораторів та DTO. Надає інтерактивний інтерфейс для тестування
ендпоінтів.

Pino -- швидкий логер для Node.js. Використовується для логування подій,
помилок та налагодження. Забезпечує структуроване логування, високу
продуктивність та підтримку різних рівнів логування.

Nodemailer -- бібліотека для відправки електронних листів.
Використовується для надсилання верифікаційних листів, повідомлень про
помилки та інших сповіщень користувачам. Підтримує різні транспорти та
шаблонізацію листів.

AWS SDK для S3 -- бібліотека для роботи з Amazon S3. Використовується
для зберігання медіафайлів (зображення, відео) через S3-сумісне сховище
MinIO. Забезпечує завантаження, завантаження та управління файлами в
хмарному сховищі.

Telegram API -- офіційна бібліотека для роботи з Telegram Bot API.
Використовується для збору даних з Telegram-каналів через офіційний API.
Забезпечує доступ до публічних каналів та обробку повідомлень.

class-validator та class-transformer -- бібліотеки для валідації та

вхідних даних через декоратори та перетворення об\'єктів між форматами.
Інтегруються з NestJS для автоматичної валідації запитів.

Jest -- фреймворк для тестування JavaScript та TypeScript коду.
Використовується для написання юніт-тестів, інтеграційних тестів та e2e
тестів. Забезпечує покриття коду, моки та асинхронне тестування.

ESLint -- інструмент для статичного аналізу коду. Використовується для
виявлення помилок, дотримання стилю коду та найкращих практик.
Налаштований для TypeScript, NestJS та Vue.js з автоматичним
виправленням помилок.

Prettier -- інструмент для форматування коду. Використовується для
автоматичного форматування TypeScript, JavaScript, JSON та інших файлів.
Забезпечує консистентний стиль коду в команді розробників.

Docker -- платформа для контейнеризації додатків. Використовується для

упаковувати додаток разом з залежностями та забезпечує однакову
поведінку на різних платформах.

Drizzle Kit -- інструмент для роботи з міграціями бази даних у Drizzle

instaloader -- Python-бібліотека для збору даних з Instagram.

feedparser -- Python-бібліотека для парсингу RSS та Atom стрічок.

twscrape -- Python-бібліотека для збору даних з Twitter/X.

Pydantic -- бібліотека для валідації даних у Python. Використовується в

VeeValidate -- бібліотека для валідації форм у Vue.js. Використовується

## Аналіз безпеки даних

- Аналіз безпеки даних. Система включає заходи безпеки для захисту даних
  користувачів, запобігання несанкціонованому доступу та забезпечення
  цілісності інформації. Нижче наведено аналіз вразливостей та
  реалізовані механізми безпеки.

- Захист від SQL-ін\'єкцій. Для запобігання SQL-ін\'єкціям
  використовується Drizzle ORM, який забезпечує параметризовані запити
  до бази даних. Всі запити до PostgreSQL виконуються через типобезпечні
  методи ORM, що автоматично екранує спеціальні символи та перешкоджає
  впровадженню зловмисного SQL-коду. Drizzle ORM генерує безпечні
  запити, де всі параметри передаються окремо від структури запиту, що
  робить неможливим виконання довільного SQL-коду через вхідні дані.

- Захист від XSS-атак. Для запобігання міжсайтовому скриптингу (XSS)
  використовується валідація та санітизація вхідних даних. На клієнті
  застосовується бібліотека isomorphic-dompurify для очищення
  HTML-контенту від потенційно небезпечних елементів перед
  відображенням. На сервері всі вхідні дані валідуються через
  class-validator з декораторами, що перевіряють формат, довжину та тип
  даних. Додатково використовується функція stripHtmlTags для видалення
  HTML-тегів з тексту перед збереженням у базі даних, що запобігає
  збереженню та виконанню зловмисних скриптів.

- Аутентифікація та авторизація. Система використовує JWT (JSON Web
  Tokens) для аутентифікації користувачів. Access token передається
  через заголовок Authorization та має обмежений час дії для зменшення
  ризику компрометації. Refresh token зберігається в HTTP-only cookie,
  що захищає його від доступу через JavaScript та запобігає XSS-атакам
  на токени. Перед доступом до захищених ресурсів виконується перевірка
  валідності обох токенів через AuthGuard, який також перевіряє
  наявність необхідних ролей користувача для доступу до конкретних
  ендпоінтів.

- Хешування паролів. Паролі користувачів зберігаються у хешованому
  вигляді за допомогою алгоритму bcrypt з коефіцієнтом складності 10. Це
  забезпечує стійкість до атак перебором та робить неможливим
  відновлення оригінального пароля навіть при компрометації бази даних.
  При авторизації порівняння пароля виконується через безпечну функцію
  bcrypt.compare, яка захищена від атак за часом виконання.

- Валідація вхідних даних. Всі вхідні дані валідуються на рівні DTO
  через декоратори class-validator перед обробкою бізнес-логікою.
  ValidationPipe в NestJS автоматично перевіряє всі запити та повертає
  помилки валідації для некоректних даних. Додатково використовується
  Zod для валідації конфігурації та складних структур даних. Валідація
  URL джерел виконується через спеціалізований сервіс, який перевіряє
  формат, домен та структуру URL перед додаванням до системи.

- Налаштування CORS. CORS налаштований для дозволу запитів лише з
  дозволеного домену клієнта, що запобігає несанкціонованим запитам з
  інших доменів. Дозволено лише необхідні HTTP-методи та заголовки, а
  credentials встановлено в true для підтримки авторизації через
  cookies. Це забезпечує захист від атак типу CSRF та обмежує можливість
  використання API з неавторизованих джерел.

- Безпека зберігання даних. Чутливі дані, такі як паролі та токени,
  зберігаються у зашифрованому або хешованому вигляді. Паролі ніколи не
  зберігаються у відкритому вигляді, а токени оновлення зберігаються в
  базі даних з інформацією про час закінчення дії. Медіафайли
  зберігаються в S3-сумісному сховищі MinIO з контролем доступу через
  префікси та права доступу. База даних налаштована з обмеженнями
  доступу та використовує транзакції для забезпечення цілісності даних.

- Верифікація електронної пошти. Перед повним доступом до системи
  користувачі повинні верифікувати свою електронну пошту через
  JWT-токен, надісланий у листі. Це запобігає створенню фейкових
  акаунтів та забезпечує валідність контактних даних користувачів. Токен
  верифікації має обмежений час дії та може бути використаний лише один
  раз, що зменшує ризик компрометації.

- Обробка помилок та логування. Система використовує централізований
  фільтр обробки помилок, який не розкриває детальну інформацію про
  внутрішню структуру системи у відповідях клієнтам. Логування
  виконується через структурований логер Pino, який записує події без
  чутливих даних, таких як паролі або повні токени. Це запобігає витоку
  інформації через логи та забезпечує можливість аналізу безпеки без
  компрометації даних.

- Захист від несанкціонованого доступу. Доступ до адміністративних
  функцій обмежений користувачам з роллю admin через перевірку ролей у
  AuthGuard. Кожен захищений ендпоінт перевіряє наявність необхідних
  прав доступу перед виконанням операції. Користувачі можуть редагувати
  та видаляти лише власні ресурси, такі як статті, що запобігає
  несанкціонованій модифікації чужих даних.

- Безпека передачі даних. Система розроблена з розрахунком на
  використання HTTPS у продакшн-середовищі для шифрування даних під час
  передачі між клієнтом та сервером. Cookies з refresh токенами
  налаштовані з прапорцями secure та sameSite для захисту від
  перехоплення та CSRF-атак. У продакшн-середовищі прапорець secure
  автоматично встановлюється в true, що вимагає використання HTTPS для
  передачі cookies.

- Захист від перебору паролів. Система не надає детальну інформацію про
  причину невдалої авторизації, повертаючи загальне повідомлення про
  некоректні дані. Це ускладнює атаки перебором, оскільки зловмисник не
  може визначити, чи існує користувач з вказаним email. Додатково можна
  реалізувати обмеження кількості спроб авторизації з одного IP-адреси
  для подальшого захисту.

Висновки

Реалізовані механізми безпеки забезпечують захист від основних типів
атак, таких як SQL-ін\'єкції, XSS, несанкціонований доступ та витік
чутливих даних. Використання сучасних практик безпеки, таких як
хешування паролів через bcrypt, JWT-токени з обмеженим часом дії,
валідація вхідних даних та налаштування CORS, створює надійний захист
для веб-сервісу. Для подальшого покращення безпеки рекомендовано додати
rate limiting для API, двокрокову авторизацію та регулярні аудити
безпеки.

## Висновки до розділу

У розділі 3 виконано конструювання та розроблення програмного
забезпечення для веб-сервісу формування новин на основі публічних
сторінок у соціальних мережах.

принципів Domain-Driven Design. Архітектура представлена через діаграми
C4 Model на трьох рівнях: System Context, Container та Component.
Система включає веб-додаток на Nuxt 3, серверний додаток на NestJS та
базу даних PostgreSQL.

Підпункт 3.2 присвячено важливим архітектурним рішенням: інтеграція із
зовнішніми системами через веб-скрапінг та HTTP-запити, асинхронна
обробка через BullMQ, кешування через Redis, захист через Drizzle ORM та
контейнеризація через Docker.

У підпункті 3.3 обґрунтовано вибір засобів розробки. Проведено
порівняльний аналіз альтернатив, що підтвердив оптимальність обраного
стеку.

Підпункт 3.4 містить конструювання програмного забезпечення: описано

оркестрація збору даних, витягування ідентифікаторів з URL, асинхронна
обробка завдань), розроблено структури даних та програмні модулі,
створено базу даних на PostgreSQL з десятьма основними таблицями та
вісьмома переліками типів, описано утиліти та бібліотеки розробки.

Підпункт 3.5 містить аналіз безпеки даних: розглянуто захист від
SQL-ін\'єкцій, XSS-атак, реалізовано аутентифікацію через JWT-токени,
хешування паролів через bcrypt, валідацію вхідних даних, налаштування
CORS та інші механізми безпеки.

Розділ 3 створив архітектуру та реалізацію веб-сервісу, включаючи вибір
технологій, розробку алгоритмів, проектування бази даних та забезпечення
безпеки, що забезпечує основу для подальшого тестування та розгортання
програмного забезпечення.
