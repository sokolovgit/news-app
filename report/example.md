# НАЙМЕНУВАННЯ ТА ГАЛУЗЬ ЗАСТОСУВАННЯ

Назва розробки: Веб-застосунок підтримки роботи рієлторської контори.

Галузь застосування: управління оголошеннями нерухомості.

Наведене технічне завдання поширюється на розробку веб-застосунку AtHome
\[AT\], котрий використовується для створення, публікації, пошуку та
адміністрування оголошень щодо купівлі, продажу та оренди нерухомості і
призначений для рієлторських компаній, агентів з нерухомості та осіб,
які зацікавлені в придбанні або оренді житлової нерухомості.

# ПІДСТАВА ДЛЯ РОЗРОБКИ

Підставою для розробки «AtHome» є індивідуальний навчальний план
студента та РНП затверджена кафедрою інформатики та програмної інженерії
Національного технічного університету України «Київський політехнічний
інститут імені Ігоря Сікорського».

# ПРИЗНАЧЕННЯ РОЗРОБКИ

Розробка призначена для підвищення зручності та якості взаємодії
потенційних клієнтів з агентами з нерухомості завдяки автоматизації
створення оголошень і впровадженню зручної системи фільтрації для пошуку
релевантних пропозицій.

Метою розробки є підвищення зручності та швидкості обробки інформації
про об'єкти нерухомості завдяки використанню фільтрів і технологій
штучного інтелекту при створенні та пошуку оголошень. Розроблюваний
вебзастосунок забезпечить зручне управління оголошеннями через
інструменти адміністратора системи та дозволить швидко розміщувати,
редагувати й знаходити пропозиції нерухомості. Це сприятиме економії
часу як для рієлтора, так і для клієнта та поліпшить якість взаємодії
між ними.

У даній курсовій роботі представлено розробку веб-застосунку для
підтримки діяльності рієлторської контори. Основною особливістю даного
застосунку є впровадження можливостей штучного інтелекту для

нерухомості та створено модуль із використанням технологій штучного

# ПЕРЕДПРОЄКТНЕ ОБСТЕЖЕННЯ ПРЕДМЕТНОЇ ОБЛАСТІ

## **Аналіз предметної області** 

Веб-застосунок - це програмне забезпечення, яке функціонує через
Інтернет і доступне через веб-браузер. Воно складається з двох основних
компонентів: клієнтської частини, що виконується у браузері користувача,
та серверної частини, яка обробляє запити і надає відповіді \[1\].

Веб-застосунки знаходять застосування у різних галузях, таких як бізнес,
освіта, розваги тощо. Їхній рівень складності та функціональність можуть
варіюватися від простих рішень до комплексних систем \[2\].

Для створення веб-застосунка рієлторської контори важливо врахувати
мінімальні вимоги, які можуть варіюватися залежно від специфіки ринку та
регуляторних норм. В сучасних IT продуктах, що створені для підтримки
роботи рієлторських контор застосовується велика кількість функцій, що
дозволяє завершувати повний цикл продажу або оренди житла онлайн, без
необхідності взаємодії рієлторів та орендаторів наживо. Наприклад, в
просунутих системах імплементовані можливості завантаження та підписання
документів електронним підписом, що дозволяє оформити договір о
купівлі/оренді житла онлайн.

Відповідно до цього мінімальними обов'язковими критеріями для створення
веб застосунку \[3\] \[4\], що буде підтримувати діяльність рієлторської
контори, є користувацький, рієлторський та адміністративний інтерфейси,
для роботи з нерухомістю. Пошукові фільтри, що дозволяють знаходити
нерухомість за обраними критеріями.

Багато сучасних веб-застосунків для рієлторської діяльності мають низку
функціональних обмежень, які ускладнюють їх використання. Однією з
головних проблем є низький рівень автоматизації рутинних завдань, таких
як створення описів об\'єктів \[5\], обробка клієнтських запитів чи
організація зустрічей \[6\], що збільшує навантаження на рієлторів.

Одним із ключових напрямків покращення є автоматизація рутинних задач,
зокрема створення описів об'єктів нерухомості за допомогою штучного
інтелекту, значно скоротить час на виконання цих процесів.

Таким чином, метою курсової роботи є створення зручного та
функціонального інструменту, який не лише повторює перевірені підходи,
але й додає інноваційний елемент автоматизації у вигляді AI-модуля.

### Аналіз відомих алгоритмічних та технічних рішень

У процесі розробки веб-застосунку для підтримки роботи рієлторської
контори було проаналізовано відомі алгоритмічні та технічні рішення, які
застосовуються для створення веб-сайтів такого типу. Зокрема, увагу було
приділено вибору також архітектурних підходів, що дозволяють забезпечити
ефективність, продуктивність і

Архітектурні підходи

Для створення веб-застосунків рієлторського типу зазвичай використовують
різні архітектурні підходи, залежно від вимог до продуктивності,
масштабованості та зручності розробки. Серед основних архітектурних
підходів варто виділити наступні:

Монолітна архітектура

Монолітна архітектура передбачає розробку програми як єдиного цілісного
блоку, де всі компоненти взаємодіють безпосередньо. Вона найчастіше
використовується для невеликих і середніх за складністю проєктів через
простоту реалізації.

Мікросервісна архітектура

Мікросервісна архітектура будується на розділенні системи на незалежні
сервіси, кожен з яких виконує окрему функцію. Цей підхід дозволяє легко
масштабувати систему та інтегрувати нові модулі.

Клієнт-серверна архітектура

Цей підхід передбачає поділ системи на клієнтську частину (фронтенд) і
серверну частину (бекенд), які взаємодіють через мережу.

Архітектура з використанням серверів обробки подій (Event-driven
architecture)

Цей підхід базується на обробці подій у реальному часі, що дозволяє
створювати динамічні та реактивні системи.

Сервіс-орієнтована архітектура (SOA)

SOA базується на використанні незалежних сервісів, які можуть
взаємодіяти один з одним через чітко визначені інтерфейси.

Платформи

Розгортання веб-застосунку є важливим етапом його створення, адже саме
тоді ваш продукт стає доступним для користувачів. Сьогодні існує безліч
алгоритмічних і технічних рішень для цієї задачі, кожне з яких має свої
особливості, переваги та обмеження.

Одним із найпростіших способів розгортання є використання хмарних
сервісів, таких як Heroku \[7\], Vercel \[8\] або Netlify \[9\]. Вони
дозволяють розробникам уникнути складної конфігурації серверів та
пропонують автоматизоване розгортання.

Для масштабніших проєктів варто звернути увагу на великі хмарні
платформи, такі як Amazon Web Services (AWS) \[10\], Google Cloud
Platform (GCP) \[11\] або Microsoft Azure \[12\]. Ці сервіси
забезпечують неймовірну гнучкість і можливість масштабування вашого
застосунку, пропонуючи сотні інструментів, включно з серверами, базами
даних та хмарним зберіганням. Наприклад, AWS часто обирають завдяки його
широким можливостям, а GCP стає популярним серед команд, які займаються
машинним навчанням чи аналітикою даних.

Для більшого контролю над конфігурацією свого середовища, варто
розглянути віртуальні приватні сервери (VPS), такі як DigitalOcean
\[13\], Linode \[14\] и Hetzner \[15\]. Ці рішення дозволяють самостійно
налаштовувати сервери, але водночас вимагають базових знань системного
адміністрування.

Для тих, хто працює з Python або Django \[23\], існують спеціалізовані
сервіси, наприклад, PythonAnywhere \[16\] або Railway \[17\]. Вони
пропонують спрощену інтеграцію з цими технологіями, що особливо корисно
для швидкого старту проєкту.

Якщо застосунок складний і потребує ізоляції середовищ, можна
скористатися контейнеризацією за допомогою Docker \[18\]. Використовуючи
Docker разом із системами оркестрації, такими як Kubernetes \[19\], ви
отримуєте повний контроль над середовищем і можливість масштабувати
застосунок залежно від потреб. Хоча це рішення потребує певних знань у
DevOps, воно ідеально для великих команд і проєктів.

Застосунок можна розгорнути і на власному сервері. Це найгнучкіше
рішення, яке дає повний контроль над усіма аспектами роботи, але воно
вимагає серйозної технічної експертизи.

Зважаючи на те, що дана курсова робота є навчальним проєктом, а
технології розгортання веб-застосунків (Docker, Kubernetes чи інші
засоби оркестрації) ще не були опановані в рамках університетського
курсу, у межах роботи я здійснюю реалізацію системи у середовищі
локального сервера Next.js. Такий підхід є цілком виправданим на
поточному етапі навчання, оскільки дозволяє зосередитися на основній
функціональності веб-застосунку, його структурі та якості реалізації.

Разом із цим я передбачаю можливість подальшого розширення та адаптації
архітектури проєкту для перенесення на Kubernetes чи інші платформи
контейнеризації у майбутньому. Це стане логічним наступним кроком після
опанування відповідних технологій у навчальному процесі. Наразі ж
виконання застосунку в середовищі локального Next.js забезпечує
необхідну працездатність і відповідає навчальним цілям курсової роботи.

Опис послідовності реєстрації користувача.

- гість відкрив вікно для реєстрації в системі;

- гість вводить свої дані;

- система валідує дані;

- коли введені дані розпізнано як валідні, система надсилає лист на
  пошту користувача для підтвердження реєстрації і сповіщує про це
  користувача;

- гість переходить на пошту;

- гість натискає на посилання у листі від системи;

- система отримує підтвердження на створення акаунту;

- система повертає користувача на екран входу до системи (/login).

Опис послідовності додання нерухомості.

1)  Адміністратор входить у адмін панель та натискає на кнопку \"ADD\" у
    розділі estate.

2)  Система відкриває сторінку для заповнення та чекає на отримання
    даних.

3)  Адміністратор вводить інформацію у формі, що відкрилась, додає
    медіа.

4)  Система валідує дані.

5)  Адміністратор заповнює опис:

    1)  Адміністратор використовує кнопку генерації опису за допомогою
        ШІ (або).

        - Система генерує опис

        - Адміністратор може редагувати опис за потреби.

    2)  Адміністратор не використовує кнопку генерації опису (або).

        - Адміністратор вводить опис власноруч.

6)  Адміністратор підтверджує створення оголошення.

7)  Система зберігає створене оголошення.

Опис послідовності написання повідомлення рієлтору (включає в себе
загальний алгоритм пошуку нерухомості).

1)  Користувач заходить на головну сторінку.

2)  Користувач шукає нерухомість.

3)  Користувач застосовує фільтрацію.

4)  Користувач обирає нерухомість з доступних варіантів.

5)  Система виконує переадресацію на сторінку нерухомості.

6)  Користувач заповнює свої дані в поля email, full name, phone,
    message.

7)  Cистема валідує дані.

8)  Користувач натискає кнопку для надсилання повідомлення.

9)  Система надсилає повідомлення рієлтору.

## **Висновки до розділу**

У даному розділі було визначено основні аспекти роботи рієлторської
контори та сформовано ключові функціональні вимоги до веб-застосунку.
Проаналізовано популярні платформи, такі як Sreality та DOM.RIA, їх
функціональні можливості, переваги та недоліки. Виявлено, що жодна з них
не інтегрує штучний інтелект для автоматичного створення описів
об'єктів, що підкреслює інноваційність розроблюваного рішення. Окрему
увагу приділено аналізу сучасних підходів до розгортання
веб-застосунків. Оскільки курсова робота є навчальним проєктом, а
технології контейнеризації та оркестрації ще не вивчалися в рамках
навчальної програми, для реалізації обрано найоптимальніший підхід на
даному етапі --- запуск системи на локальному сервері Next.js. Це
дозволяє зосередитися на функціональності, архітектурі та логіці
застосунку без ускладнення процесу розробки додатковими інструментами
розгортання. У майбутньому, після опанування відповідних технологій,
проєкт може бути адаптований до розгортання у контейнеризованому
середовищі. У межах цього етапу також було описано основні
бізнес-процеси рієлторської діяльності, на основі чого сформовано модель
бізнес-процесів у вигляді BPMN-діаграм, що деталізують ключові сценарії
взаємодії користувачів із системою, зокрема процеси реєстрації,
додавання об'єктів нерухомості та надсилання повідомлень рієлторам.

Цей розділ забезпечив глибоке розуміння предметної області, підкріплене
аналізом існуючих рішень, та заклав основу для розробки веб-застосунку.
Визначення вимог до функціоналу, технологічних інструментів і
архітектурних підходів гарантує, що створюваний продукт відповідатиме
сучасним вимогам ринку нерухомості, а включення модуля на основі
штучного інтелекту дозволить значно заощадити час рієлторам.

## **Розроблення нефункціональних вимог**

### Доступність:

- система має забезпечувати максимально допустимий час простою не більше
  15 хвилин на місяць (Downtime budget ≤ 0.035%);

- середній час відновлення (MTTR) не повинен перевищувати 10 хвилин;

- середній час між відмовами (MTBF) має бути не менше 72 годин.

### Продуктивність:

- система має підтримувати щонайменше 100 одночасно активних
  користувачів без погіршення швидкодії;

- середній час реакції інтерфейсу користувача - до 500 мс;

- час обробки запитів на сервері - не більше 1 секунди у 95% випадків;

- швидкість рендерингу сторінки - не більше 2 секунд на клієнтській
  частині (при швидкості інтернету 20 Мбіт/с);

- максимальна кількість транзакцій - щонайменше 5000 на добу без
  деградації системи.

### Обмеження:

- реалізація має бути виконана з використанням мови JavaScript згідно зі
  стандартом ECMAScript 2024. Для клієнтської та серверної частин слід
  використовувати фреймворк Next.js версії 15.5.4 \[35\], що базується
  на бібліотеці React версії 19.1 \[24\].;

- система повинна працювати у сучасних браузерах, включаючи Google
  Chrome 110+, Mozilla Firefox 110+, Microsoft Edge 110+, Safari 16+;

- система має функціонувати на пристроях із мінімальною роздільною
  здатністю 360×640 пікселів.

### Вимоги до безпеки:

- має бути реалізоване розмежування доступу за ролями: адміністратор,
  рієлтор, користувач, гість;

- дані користувача (email, пароль) мають оброблятися згідно з вимогами
  GDPR;

- усі з'єднання між клієнтом і сервером повинні бути захищені через
  HTTPS;

- повинна бути реалізована перевірка валідності даних на клієнті й
  сервері;

- вхід у систему має бути захищеним (використання токенів, обмеження
  спроб авторизації);

- час зберігання персональних даних у БД - до моменту видалення акаунту.

### Масштабованість:

- має бути передбачена можливість вертикального масштабування шляхом
  підвищення характеристик серверу;

- має бути забезпечена підтримка горизонтального масштабування для
  клієнтської частини через CDN, а для серверної - через
  Docker-контейнери \[18\].

### Конфігурованість:

Вимоги до конфігурованості не висуваються.

### Вимоги до зручності використання:

Максимальна глибина вкладення у веб-застосунку -- не більше чотирьох
рівнів.

достовірної інформації щодо апаратних характеристик, які використовують
аналоги на кшталт Sreality \[20\] або DOM.RIA \[21\], було прийнято
рішення спиратися на узагальнені поради щодо налаштування середовища для
сучасної СУБД PostgreSQL та популярних стеків веб-розробки.

Вибір процесора базується на необхідності забезпечити стабільну
паралельну обробку запитів від користувачів та сервісів. Мінімальні
характеристики у 2 ядра з частотою 2.0-2.5 ГГц достатні для запуску
системи, тоді як рекомендовані параметри у 4 ядра з частотою 2.9-3.5 ГГц
дозволяють забезпечити комфортну роботу під навантаженням та у випадках
активного використання функцій на стороні бекенду. Обсяг оперативної
пам'яті у 4-8 ГБ у мінімальній конфігурації дозволяє запуск системи з
обмеженим набором користувачів, але для оптимальної роботи серверу з
базою даних PostgreSQL та кешуванням запитів рекомендується не менше 16
ГБ ОЗП.

Щодо дискової підсистеми, мінімальний SSD обсягом 50-100 ГБ обраний з
урахуванням базової потреби у збереженні даних користувачів, оголошень
та зображень. Натомість SSD NVMe з обсягом понад 200 ГБ рекомендовано
через вищу швидкість читання-запису, що важливо для роботи з великими
масивами даних, зокрема в умовах зростання бази. Швидкість мережі у 100
Мбіт/с є достатньою для базового обміну даними, однак для стабільної
роботи застосунку за умови активного використання клієнтами краще
використовувати канал зі швидкістю 1 Гбіт/с.

Щодо операційної системи, вибір зроблено на користь Ubuntu \[25\],
оскільки вона є однією з найпоширеніших платформ для серверного
розгортання Next.js-застосунків \[35\], і повністю підтримує роботу з
PostgreSQL. Мінімальні вимоги передбачають Ubuntu 20.04+, тоді як для
сумісності з новими версіями бібліотек і оновлень безпеки рекомендовано
використовувати Ubuntu 22.04+. Щодо СУБД, PostgreSQL 12 є стабільною
базою для запуску застосунку, однак у рекомендованих налаштуваннях
зазначено PostgreSQL 14+, оскільки ці версії мають покращену
продуктивність, розширені можливості індексації, сучасні механізми
обробки транзакцій та активну підтримку з боку розробників.

## **Постановка завдання на розробку програмного забезпечення**

Метою розробки є створення сучасного веб-застосунку, який забезпечить
підтримку ключових бізнес-процесів рієлторської контори, автоматизує
рутинні операції та підвищить ефективність взаємодії з клієнтами.
Програмне забезпечення має бути доступним з будь-якого пристрою з
інтернет-доступом, мати зручний інтерфейс для користувачів і рієлторів,
а також включати модуль штучного інтелекту для автоматичного формування
описів нерухомості.

Основна ціль полягає в покращенні якості обслуговування клієнтів і
зменшенні витрат часу на рутинну діяльність завдяки цифровізації
процесів додавання, пошуку та управління об'єктами нерухомості.

Для досягнення мети необхідно вирішити такі задачі:

- реалізувати механізми реєстрації та авторизації.

- створити інтерфейси для додавання, редагування, перегляду та видалення
  об'єктів нерухомості;

- впровадити систему пошуку з можливістю фільтрації;

- розробити модуль генерації текстових описів об'єктів;

- забезпечити засоби комунікації між користувачем і рієлтором (контактна
  форма);

- створити адміністративну панель для управління контентом та запитами;

- передбачити масштабованість, безпеку та відповідність сучасним вимогам
  до веб-застосунків.

Таким чином, кінцевим результатом має стати стабільний, безпечний і
масштабований веб-застосунок, який забезпечить підтримку основних
процесів агентства нерухомості.

## **Висновки до розділу**

У другому розділі було сформульовано та деталізовано вимоги до
програмного забезпечення, необхідного для ефективного функціонування
веб-застосунку рієлторської контори. На основі аналізу предметної
області та постановки завдання визначено ключові сценарії використання
системи, які реалізуються через низку use-case моделей для різних
категорій користувачів: клієнтів, рієлторів та адміністраторів.

Здійснено розроблення функціональних вимог, які охоплюють базовий і
розширений функціонал системи, включаючи авторизацію, фільтрацію
об'єктів, генерацію описів за допомогою ШІ, управління контентом,
адміністративні дії тощо. Усі функції супроводжуються відповідними
сценаріями взаємодії та структурованими таблицями.

Окрему увагу приділено нефункціональним вимогам, які визначають вимоги
до продуктивності, доступності, масштабованості, безпеки та сумісності
програмного забезпечення. Наведено обґрунтовані системні вимоги до
серверного середовища, базуючись на сучасних стандартах і практиках у
веб-розробці.

За результатами розділу сформовано технічне завдання на розробку
програмного забезпечення.

## **Архітектура програмного забезпечення**

Під час розробки застосунку було використано гібридну архітектурну
модель, що поєднує елементи клієнт-серверної взаємодії та серверного
рендерингу. Узагальнене схематичне зображення архітектури представлено
на рисунку 3.1.

Ключові компоненти клієнт-серверної архітектури включають:

Клієнт (frontend) - відповідає за надання користувачу зручного
інтерфейсу для взаємодії з системою. Він забезпечує інтерактивність,
реагує на події та використовує функціональні можливості браузера.

Сервер (backend) - обробляє запити, що надходять від клієнта. Цей
компонент реалізує логіку, яка не потребує безпосереднього контакту з
користувачем, зокрема доступ до бази даних.

- Overpass API \[32\] - для отримання об'єктів поряд з нерухомістю;

- AWS S3 \[33\] - для збереження та оптимізації медіафайлів,

## **Архітектурні рішення та обґрунтування вибору засобів розробки**

Для реалізації веб-застосунку було обрано стек технологій, до якого
входить JavaScript з фреймворком Next.js 15 \[35\] для серверної і
клієнтської частини, а також реляційна система управління базами даних
PostgreSQL \[22\]. Такий вибір обумовлений необхідністю швидкої
розробки, ефективного поєднання серверної логіки та інтерфейсу
користувача в одному середовищі, а також забезпечення масштабованості у
подальшому. Next.js надає гібридну архітектуру, що поєднує серверні
компоненти, API маршрути та клієнтські інтерфейси в єдиній структурі
проєкту. Це спрощує розгортання, покращує продуктивність та полегшує
інтеграцію із зовнішніми сервісами. Компонентний підхід React \[24\], що
лежить в основі Next.js, забезпечує високу продуктивність, модульність
та гнучкість при розробці інтерфейсу.

Хоча у тексті для стислості використовується термін \"JavaScript\",
проєкт буде реалізовано мовою TypeScript \[36\]. TypeScript забезпечує
статичну типізацію, кращу передбачуваність коду і знижує кількість
помилок під час розробки, але в кінцевому підсумку компілюється у
JavaScript, тому концептуально обидві мови розглядаються в єдиному
контексті.

Управління користувачами реалізовуватиметься через власну систему
реєстрації та авторизації. Однофакторна аутентифікація виконуватиметься
на основі токенів, що забезпечуватимуть безпечну роботу з сесіями без
необхідності використання сторонніх сервісів типу Single Sign-On.
Зберігання акаунтів користувачів відбуватиметься у базі даних. Усі
паролі шифруватимуться згідно з вимогами безпеки.

Інтеграція із зовнішніми системами буде реалізована через
REST-протоколи. Для надсилання електронних листів використовуватиметься
Resend API \[30\], що дозволить надійно доставляти підтвердження
реєстрації та повідомлення від користувачів до рієлторів. Генерація
описів нерухомості здійснюватиметься за допомогою OpenAI API \[31\],
який забезпечуватиме обробку природної мови. Для отримання об'єктів
поблизу буде використовуватися Overpass API \[32\], а для зберігання
медіафайлів використовуватиметься AWS S3 \[33\].

В якості типу бази даних обрано реляційну модель, оскільки дані
нерухомості мають чітку структуру, підлягають нормалізації і пов\'язані
між собою. PostgreSQL \[22\] забезпечує надійну роботу з транзакціями,
підтримку індексації та хорошу продуктивність на середніх і великих
об\'ємах даних. Альтернативні рішення на основі NoSQL не
використовуватимуться, оскільки у нашому випадку немає переваг у відмові
від табличної структури.

У порівнянні з іншими веб-технологіями, такими як PHP з Laravel \[26\]
або Java зі Spring \[27\], Next.js має нижчий поріг входу, працює на
JavaScript і дозволяє швидко створити робочий прототип завдяки поєднанню
серверної і клієнтської логіки в одному фреймворку. Next.js забезпечує
вищу швидкість розробки, ніж традиційні бекенд фреймворки, оскільки не
потребує окремої інфраструктури для API і дає можливість реалізувати
серверні маршрути безпосередньо в структурі проєкту.

Для побудови інтерфейсу використовується React \[24\], що є основою
Next.js. React обрано замість Vue \[29\] або Angular \[28\] через широку
підтримку спільноти, зручну інтеграцію з будь-якими API, високу
гнучкість компонентної моделі та ефективну взаємодію з серверною
частиною фреймворку. React має низький поріг входу і дає можливість
швидко створювати масштабовані, адаптивні та продуктивні інтерфейси.

Для розробки клієнтської та серверної логіки використовуватиметься
Visual Studio Code \[34\], який вирізняється легкістю, модульністю та
великою кількістю корисних розширень для роботи з JavaScript, TypeScript
та Next.js.

## **Конструювання програмного забезпечення**

У процесі розробки вебзастосунку було використано структурний підхід,
притаманний фреймворку Next.js \[35\], який організовує логіку за
принципами Route-based та Feature-based архітектури. Такий підхід
дозволяє розмежувати відповідальності між окремими функціональними
модулями, спростити масштабування та зробити проєкт більш зрозумілим для
подальшої підтримки.

Уся клієнтська та серверна логіка організована навколо каталогу app,
який формує маршрути застосунку. Кожен маршрут являє собою окремий
модуль і може містити власні компоненти, серверні дії, стилі та вкладені
підмаршрути. Таким чином забезпечується природна ізоляція функціональних
областей.

### App Router та Feature-based структура

Структура каталогу app (рисунок 3.17) побудована відповідно до принципу
feature-based архітектури, де кожна ключова область бізнес-логіки має
свій власний простір:

- (auth) - модуль автентифікації з окремими сторінками login, register
  та verify;

- admin - модуль адміністративної панелі, що містить підрозділи для
  керування агентами, користувачами та об\'єктами нерухомості;

- estate - модуль роботи з нерухомістю, що включає сторінку додавання,
  детальне відображення об\'єкта та допоміжні компоненти;

- profile - модуль профілів користувачів і збережених оголошень;

- search - модуль пошуку нерухомості, який містить інструменти
  фільтрації, карту та сторінку результатів.

Кожен модуль містить власну підструктуру components, яка включає лише ті
компоненти, що стосуються конкретної функціональної області. Така
ізоляція полегшує обслуговування, повторне використання та тестування.

У рамках розробки вебзастосунку було створено низку сутностей, які
відображають ключові компоненти предметної області ринку нерухомості.
Основні сутності включають User, Account, Authenticator, Estate,
EstateApartment, EstateHouse, EstateMedia, EstateTranslation,
SavedEstate та допоміжні таблиці для багатозначних виборів
(HeatingSource, Water, Electricity, InternetConnection та інші). Для
зручності роботи з категоріями, станами та типами було створено велику
кількість ENUM-типів у PostgreSQL, які описують види нерухомості,
енергетичні класи, категорії будівель, комунікації, стан будинку, види
водопостачання, електрики, телекомунікацій тощо.

Більшість сутностей пов\'язана між собою через відношення OneToMany або
ManyToMany, що дозволяє гнучко описати складну модель нерухомості з
багатьма ознаками. Наприклад, одна нерухомість може мати багато
медіафайлів, багато записів про комунікації та багато доступних
варіантів опалення, а один користувач може зберігати кілька оголошень
або публікувати різні об\'єкти як агент.

Для роботи з даними використано ORM-бібліотеку Drizzle ORM \[37\], яка
виконує автоматичне формування запитів, валідацію типів на рівні
TypeScript \[36\] та інтеграцію з PostgreSQL. Усі таблиці створено за
допомогою декларативних методів Drizzle, таких як pgTable, serial,
integer, text, timestamp, boolean, doublePrecision та інші. Drizzle ORM
також надає механізм визначення зв\'язків через references, що дозволяє
надійно підтримувати зовнішні ключі та каскадні операції.

DTO класи як окремі об\'єкти не створюються, оскільки в архітектурі
Next.js з використанням серверних дій передача даних між клієнтом і
сервером здійснюється через схеми валідації на основі Zod \[38\]. Це
забезпечує ізоляцію внутрішньої структури таблиць від зовнішнього
представлення даних та дозволяє централізовано контролювати правильність
полів, що передаються в API.

Для валідації форми нерухомості, користувачів та дій адміністратора
створено окремі об\'єкти Zod у модулі zodObjects.ts, які гарантують
коректність даних перед записом у базу. Таким чином, комбінація Drizzle
ORM, Zod та статичної типізації TypeScript забезпечує безпечну та
передбачувану модель даних у застосунку.

Детальніше про Drizzle ORM, Zod, Tailwind CSS, Shadcn можна прочитати у
розділі 3.3.6 \"Опис утиліт, бібліотек та іншого ПЗ\".

## **Аналіз безпеки даних**

Під час розробки вебзастосунку особливу увагу приділено захисту
персональних даних користувачів, а також забезпеченню цілісності,
конфіденційності та доступності інформації. Архітектурні рішення та
вибрані технології забезпечують багаторівневий підхід до безпеки, що
охоплює автентифікацію, авторизацію, захист бази даних, обробку
медіафайлів, роботу з токенами та інтеграцію із зовнішніми сервісами.

#### Захист автентифікації та облікових записів

Для керування користувачами використовується бібліотека NextAuth \[35\],
яка реалізує сучасні підходи до автентифікації. Паролі користувачів не
зберігаються у відкритому вигляді - перед записом у базу даних вони
проходять хешування засобами бібліотеки bcrypt \[42\] з використанням
криптографічно стійких алгоритмів.

#### Захист бази даних та логіка доступу

База даних PostgreSQL \[22\] організована із суворим дотриманням
принципів нормалізації, що мінімізує дублювання інформації та зменшує
ризик логічних помилок. Усі сутності пов'язані через зовнішні ключі з
каскадними обмеженнями, що запобігає неконсистентності даних.

Drizzle ORM \[37\] забезпечує типобезпечні запити та автоматичне
мапування даних, що суттєво знижує ризик SQL-ін'єкцій. Усі запити
виконуються через підготовлені оператори, а параметри користувача ніколи
не вставляються в SQL вручну.

Валідація даних реалізована за допомогою Zod \[38\], що унеможливлює
потрапляння у систему некоректних або потенційно небезпечних даних.

#### Захист API та бізнес-логіки

Використання серверних дій (Next.js Server Actions) дозволяє ізолювати
бізнес-логіку від клієнтської частини. Усі критичні операції виконуються
на сервері та не розкриваються у вигляді публічних REST-ендпоінтів.

Зовнішні API (OpenAI, Overpass, Resend) викликаються лише з сервера, а
секретні ключі зберігаються у змінних середовища за допомогою dotenv.

#### Висновок

Архітектура застосунку реалізує комплексний багаторівневий підхід до
безпеки, який охоплює захист автентифікації, контроль доступу,
криптографічний захист паролів і токенів, типобезпечний доступ до бази
даних, валідацію введення, ізоляцію медіафайлів у S3, а також безпечну
інтеграцію з зовнішніми сервісами. Сукупність цих засобів забезпечує
надійний захист персональних даних користувачів та стійкість системи до
основних класів атак.

## **Висновки до розділу**

У даному розділі було представлено архітектурні рішення та структуру
вебзастосунку, побудованого на фреймворку Next.js із застосуванням
підходів Route-based та Feature-based організації. Така структура дала
змогу чітко розмежувати функціональні модулі системи - автентифікацію,
адміністративну панель, модуль нерухомості, профілі користувачів та
пошук, що спрощує масштабування та підтримку застосунку.

Було описано організацію каталогів app, lib, components, utils та
styles, кожен з яких відповідає за окремий аспект логіки або інтерфейсу.
Окремо розглянуто серверні дії, винесені у каталог lib, які реалізують
бізнес-логіку системи та забезпечують відокремлення UI від операцій
роботи з даними.

Суттєва увага приділялася моделі даних, реалізованій на основі
PostgreSQL та Drizzle ORM. У розділі докладно охарактеризовано всі
основні сутності доменної області нерухомості, їх структуру, зв'язки та
механізми цілісності даних. Додатково наведено опис допоміжних таблиць,
ENUM-типів та принципів організації схем валідації на основі Zod, що
гарантують коректність введення та передачі даних.

Розглянуто утиліти та бібліотеки, використані у проєкті: NextAuth для
автентифікації, Tailwind CSS та Shadcn для інтерфейсу, AWS SDK для
роботи з медіафайлами, OpenAI та Resend для зовнішніх інтеграцій. Їх

Засоби безпеки включають хешування паролів, типобезпечні запити Drizzle
ORM, серверні дії Next.js, валідацію даних, захист медіафайлів через S3
та ізоляцію секретних ключів у змінних середовища. У сукупності це
забезпечує цілісність, конфіденційність та стійкість системи до
більшості загроз.

Таким чином, у розділі було сформовано узагальнене бачення архітектури,
структури, інструментів та механізмів безпеки, що лежать в основі
вебзастосунку та забезпечують його надійність і подальшу

## **Опис процесів тестування**

Оскільки у вебзастосунку використано механізм Server Actions, який
забезпечує пряме виконання бізнес-логіки на сервері без явної експозиції
REST-або RPC-ендпоінтів, проведення окремого тестування API не є
необхідним. Усі взаємодії між клієнтською та серверною частинами
здійснюються через внутрішні серверні функції, що викликаються у межах
середовища Next.js та не доступні ззовні як відкриті мережеві ресурси.
Тому в рамках тестування було перевірено коректність роботи саме
серверних дій та їх інтеграцію з інтерфейсом користувача, що повністю
покриває функціональні сценарії взаємодії системи.

# РОЗГОРТАННЯ ТА СУПРОВІД ПРОГРАМНОГО ЗАБЕЗПЕЧЕННЯ

## **Розгортання програмного забезпечення**

Оскільки дана курсова робота є навчальним проєктом, а інструменти
контейнеризації та хмарного розгортання (Docker, Kubernetes, CI/CD, AWS
тощо) не входили до програми курсу, у межах цієї роботи було застосовано
локальне розгортання на вбудованому сервері Next.js.

Такий підхід дозволяє зосередитися на функціональності системи,
структурі застосунку та забезпеченні коректної роботи серверних і
клієнтських компонентів без додаткових витрат на налаштування
інфраструктури.

Next.js надає вбудоване середовище виконання, яке включає:

- серверну частину (API-маршрути та серверні компоненти);

- клієнтську частину (React-інтерфейс);

- компілятор та інструменти збірки;

- гаряче перезавантаження (Hot Reload);

- локальний вебсервер (next dev).

Таким чином, усі необхідні частини системи функціонують у єдиному
середовищі без окремого розгортання фронтенду та бекенду.

## **Локальне розгортання програмного забезпечення**

Для запуску веб-застосунку виконується стандартна процедура
ініціалізації та запуску проєкту Next.js:

Встановлення залежностей

\`\`\`

npm install

\`\`\`

Налаштування змінних середовища

У корені проєкту створюється файл .env, у якому визначаються параметри:

- підключення до бази даних PostgreSQL;

- ключі зовнішніх сервісів (Resend, OpenAI, AWS S3 тощо).

Завдяки Next.js, усі змінні середовища автоматично завантажуються під
час запуску сервера.

Запуск сервера розробки

\`\`\`

npm run dev

\`\`\`

Після запуску застосунок доступний за адресою http://localhost:3000.

У рамках локального розгортання Next.js автоматично:

- компілює серверні та клієнтські компоненти;

- забезпечує обробку API-запитів;

- взаємодіє з базою даних через Drizzle ORM;

- виконує оптимізацію рендерингу.

Такий спосіб розгортання є оптимальним для навчальних і прототипних
рішень, оскільки не потребує додаткових інструментів, надаючи повністю
працездатну систему.

## **Аналіз можливих засобів продакшн-розгортання**

Хоча фактичне розгортання в межах цієї роботи здійснюється локально,
варто коротко розглянути інструменти, які можуть бути використані для
продакшн-середовища.

Потенційними варіантами є:

- Vercel \[8\] - нативна платформа для Next.js (автоматичний деплой,
  CDN, edge-функції);

- Docker \[18\] + будь-який VPS - повний контроль над розгортанням
  застосунку;

- AWS \[10\] (ECS / EC2) - масштабованість, можливість інтеграції з RDS,
  S3 та CloudFront;

- Railway \[17\] / Netlify \[9\] - простіші альтернативи для backend-та
  frontend-хостингу.

Усі ці інструменти можуть бути використані для подальшого розвитку
системи після завершення проєкту.

## **Супровід програмного забезпечення**

Супровід системи у локальному середовищі включає:

- перевірку коректності серверних функцій Next.js;

- контроль за змінами структури бази даних (Drizzle migrations);

- оновлення залежностей проєкту;

- тестування UI та API-функціональності.

Процес оновлення у локальному середовищі є простим:

- внести зміни у код;

- система автоматично перезапускає dev-сервер;

- результати змін відображаються негайно.

Для майбутнього продакшну процедура буде передбачати:

- збірку нової версії (npm run build);

- оновлення на хостингу чи сервері;

- перевірку працездатності API та маршрутизації.

містить сучасний інтерфейс та інтегрує інноваційні технології, що
підвищують ефективність роботи користувачів.

Практична значимість роботи полягає у впровадженні функціоналу, який
дозволяє оптимізувати рутинні процеси, прискорити розміщення нових
оголошень та забезпечити зручну взаємодію між клієнтами та рієлторами.
Застосунок підтримує автоматичне генерування описів нерухомості за
допомогою штучного інтелекту, надає розгорнуту систему пошуку та
фільтрації, а також включає адміністративні інструменти для управління
користувачами та контентом.

Усі функціональні вимоги, визначені на початковому етапі проєктування,
були реалізовані. Забезпечено підтримку реєстрації та авторизації
користувачів із розмежуванням доступу, створено комплексний
CRUD-функціонал для роботи з об'єктами нерухомості, впроваджено засоби
фільтрації та пошуку, інтегровано модуль генерації описів оголошень,
